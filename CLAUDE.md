## О проекте
Совместный проект лаборатории DataMED Lab (Тель-Авивский университет, медицинский факультет) и медицинской организации Маккаби (מכבי שירותי בריאות) в области генетики. Проект возник потому, что Маккаби предложила DataMED Lab работать с генетическими данными. Принципиальное одобрение от Маккаби получено в марте 2026.

## Архитектура — две стадии

### Стадия 1 — Сбор и агрегация (ТЕКУЩАЯ)
Данные из открытых источников в интернете. Данных Маккаби на этом этапе НЕТ.

**Pipeline:**
1. **Источники данных** — список от Лины (ожидаем файл). Это справочная информация о базах данных, НЕ техническое задание. Конкретных инструкций "сделай так" — нет, нужно самому предложить архитектуру.
2. **Сбор данных:**
   - **Bulk download:** скачивание полных баз (ClinVar — весь дамп с FTP, gnomAD и др.). Обычный Python. Разово + периодическое обновление.
   - **API запросы:** точечные запросы для свежих данных. Тоже обычный Python.
   - Для сбора данных **агент НЕ нужен** — достаточно скриптов.
3. **Очистка и нормализация** — единый формат. ПРОБЛЕМА: одна мутация записана по-разному ("c.5266dupC" = "5382insC" = "rs80357906" — всё BRCA1).
4. **Chunking** — осмысленная разбивка (один вариант = один чанк с контекстом, НЕ "по 500 токенов").
5. **Векторизация** — embeddings → векторная БД.
6. **RAG-система** — поиск + генерация ответов с citations.

**Где нужен агент:** На этапе ОТВЕТА на запросы (не сбора). Врач задаёт вопрос → агент решает куда идти: векторная база → ClinVar API (свежие данные) → PubMed (новые статьи) → собирает ответ. Скрипт не умеет принимать такие решения.

### Стадия 2 — Интеграция с Маккаби (БУДУЩЕЕ, далеко)
- Данные пациентов = НЕ сырые ATCG, а результаты тестирования: "у пациента мутация rs12345 в гене Y"
- Pipeline: идентификатор мутации → поиск в RAG-базе → ответ врачу
- Приложение/интерфейс для врачей
- Возможно связано с "Maccabi HMO Cancer Panel"

## Генетика — ключевые концепции

### VUS — Variant of Uncertain Significance (ГЛАВНАЯ ПРОБЛЕМА)
40-60% всех обнаруженных вариантов. Врач не может сказать пациенту, опасна мутация или нет. Цель RAG — собрать всё известное о VUS из разных источников в одно место.

### Классификация по ACMG:
- 🟢 Benign / Likely Benign — безвредная
- 🟡 **VUS** — неясно (основная проблема)
- 🔴 Likely Pathogenic / Pathogenic — опасная

### SNP — Single Nucleotide Polymorphism
Точечное отличие в ДНК. Идентификатор: rs-номер (rs80357906).

### Фенотип vs Генотип
- Генотип — что в ДНК (код)
- Фенотип — как проявляется (результат выполнения)

### Зачем несколько источников:
| Источник | Что даёт |
|----------|----------|
| **ClinVar** | Классификация мутаций, мнения лабораторий |
| **gnomAD** | Частота в популяции (редкая → возможно опасна) |
| **PubMed** | Научные статьи, клинические доказательства |
| **dbSNP** | Технические детали (позиция, тип замены) |
| **OMIM** | Описания генов и связанных болезней |

Каждый — кусок пазла. Вместе — полная картина для врача.

## Сложности (vs обычный RAG)

1. **Хаос записей:** одна мутация = десять способов записи в разных базах
2. **Противоречия между источниками:** показывать ВСЕ мнения, НЕ удалять конфликты
3. **Осмысленный чанкинг:** генетическая запись должна быть целостной
4. **Citations обязательны:** галлюцинация в медицине = неправильная рекомендация врачу
5. **Обновления:** базы меняются (вчера VUS → сегодня Pathogenic)
